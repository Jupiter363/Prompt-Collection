# 通用 Prompt 模板库

> 适用范围：通用写作、分析、学习、代码、研究、总结、规划、评审、工具调用型任务。  
> 设计依据：综合 OpenAI Prompt Engineering / OpenAI Prompting / Anthropic Prompt Engineering Overview 的共性原则整理。  
> 核心思想：Prompt 不是“话术”，而是把任务目标、上下文、约束、输出格式、验证标准清晰传达给模型的结构化指令。

---

## 1. 通用 Prompt 结构

一个高质量 Prompt 通常包含以下 8 个部分：

```text
# 角色
你是一个{角色/专家身份}，擅长{能力范围}。

# 任务
请完成以下任务：{明确任务目标}。

# 背景
背景信息如下：
{必要上下文、业务背景、用户目标、已有材料}

# 输入材料
以下是需要处理的内容：
"""
{原始文本/代码/数据/需求/问题}
"""

# 约束条件
请遵守以下要求：
1. {约束1}
2. {约束2}
3. {约束3}

# 输出格式
请按照以下格式输出：
{Markdown / 表格 / JSON / 分点 / 报告 / 代码块 / 模板}

# 质量标准
输出需要满足：
1. {准确性标准}
2. {完整性标准}
3. {可执行性标准}
4. {风格标准}

# 自检要求
在最终输出前，请检查：
1. 是否完整回答任务；
2. 是否符合输出格式；
3. 是否存在逻辑跳跃、事实不确定或遗漏。
```

## 2. 最小可用 Prompt 模板

当任务比较简单时，可以使用这个轻量模板：

```text
请你作为{角色}，帮我完成{任务}。

背景：{简要背景}

输入：
"""
{输入内容}
"""

要求：
1. {要求1}
2. {要求2}
3. {要求3}

输出格式：{指定格式}
```

示例：

```text
请你作为一名技术面试辅导老师，帮我解释 Java ArrayList 的底层原理。

背景：我正在准备后端开发面试，希望理解原理而不是死记结论。

要求：
1. 先用通俗语言解释整体机制；
2. 再说明扩容、随机访问、插入删除的复杂度；
3. 最后给出常见面试追问。

输出格式：Markdown 分点说明。
```

## 3. 高质量通用 Prompt 模板

适合复杂任务、学习计划、技术方案、论文分析、项目设计、商业分析等场景。

```text
你是一个{领域}专家，具备{专业能力1}、{专业能力2}、{专业能力3}。

我的目标是：
{最终目标}

当前背景：
{我目前的情况、已有基础、限制条件、希望达到的效果}

请你完成以下任务：
1. {任务1}
2. {任务2}
3. {任务3}
4. {任务4}

请重点关注：
- {重点1}
- {重点2}
- {重点3}

请避免：
- {不希望出现的问题1}
- {不希望出现的问题2}
- {不希望出现的问题3}

输出要求：
1. 结构清晰，使用 Markdown；
2. 先给出总体结论，再展开细节；
3. 对关键判断给出原因；
4. 对不确定内容明确说明；
5. 给出可执行的下一步建议。
```

## 4. 任务分析类 Prompt 模板

适合让模型拆解问题、分析原因、比较方案、做决策。

```text
你是一个擅长结构化分析的{领域}顾问。

请分析以下问题：
{问题描述}

背景信息：
"""
{背景材料}
"""

请从以下角度分析：
1. 问题本质是什么；
2. 关键影响因素有哪些；
3. 可能的解决方案有哪些；
4. 每种方案的优点、缺点、适用条件；
5. 推荐方案及理由；
6. 执行时需要注意的风险。

输出格式：
- 先给结论；
- 再用表格对比；
- 最后给行动建议。

要求：
- 不要只给泛泛建议；
- 每个判断都要解释依据；
- 如果信息不足，请明确指出缺失信息，并基于当前信息做合理假设。
```

## 5. 学习路线类 Prompt 模板

适合制定学习计划、技术路线、课程规划、能力成长路径。

```text
你是一个学习路径规划专家，熟悉{领域}的知识体系、实践项目和就业/科研要求。

我的学习目标是：
{目标，例如：成为具备架构能力的 Agent 工程师}

我的当前基础：
{已有知识、薄弱点、可投入时间、项目经验}

希望学习后掌握：
1. {能力1}
2. {能力2}
3. {能力3}

请帮我制定一份系统学习路线，要求包括：
1. 阶段划分；
2. 每个阶段的学习目标；
3. 核心知识点；
4. 推荐学习资料类型；
5. 实践任务；
6. 阶段验收标准；
7. 最终可形成的项目或作品。

输出格式：
- 使用 Markdown；
- 按阶段组织；
- 每个阶段包含“学什么、怎么练、怎么验收”；
- 最后给出总体时间安排和优先级建议。

要求：
- 路线要可执行，不要只列概念；
- 要区分入门、进阶、工程化、项目实战；
- 要包含复盘和评估方法。
```

## 6. 技术方案设计类 Prompt 模板

适合系统设计、架构方案、后端项目、Agent 系统、RAG 系统、业务平台设计。

```text
你是一个资深{系统类型}架构师，擅长从业务需求出发设计可落地的技术方案。

业务背景：
{业务场景}

目标：
{系统要解决什么问题，达到什么效果}

已有条件：
{已有系统、技术栈、数据来源、限制条件}

请设计一份技术方案，内容包括：
1. 需求分析；
2. 核心业务流程；
3. 总体架构设计；
4. 核心模块拆分；
5. 数据流与调用链路；
6. 关键技术选型；
7. 异常处理与降级策略；
8. 安全与权限控制；
9. 性能与可扩展性设计；
10. 可观测性与运维监控；
11. 测试与评估方案；
12. 迭代路线。

输出要求：
- 先给整体架构概览；
- 再按模块展开；
- 对关键设计说明为什么这样选；
- 给出 MVP 版本和增强版本的区别；
- 最后给出可写进简历/论文/软著的亮点总结。
```

## 7. 写作润色类 Prompt 模板

适合论文、简历、文档、报告、邮件、演讲稿润色。

```text
你是一个擅长{写作类型}润色的专业编辑。

请帮我润色以下内容：
"""
{原文}
"""

润色目标：
1. {目标1，例如：逻辑更清晰}
2. {目标2，例如：表达更自然}
3. {目标3，例如：更符合学术/商务/面试风格}

请重点修改：
- 语句不通顺的地方；
- 逻辑跳跃的地方；
- 重复啰嗦的地方；
- 表达不够专业的地方。

请保留：
- 原文核心含义；
- 原有事实信息；
- 必要的专业术语。

输出格式：
1. 先给润色后的完整版本；
2. 再列出主要修改点；
3. 如果发现事实或逻辑问题，请单独指出。
```

## 8. 代码生成类 Prompt 模板

适合让模型写代码、补全功能、生成脚本、实现接口。

```text
你是一个资深{编程语言/技术栈}工程师。

请帮我实现以下功能：
{功能描述}

技术栈：
- 语言：{语言}
- 框架：{框架}
- 数据库/中间件：{数据库或中间件}
- 运行环境：{环境}

输入：
{输入数据或接口参数}

输出：
{预期输出或接口返回}

功能要求：
1. {功能点1}
2. {功能点2}
3. {功能点3}

工程要求：
1. 代码结构清晰；
2. 命名规范；
3. 包含必要注释；
4. 处理异常情况；
5. 给出使用示例；
6. 如有必要，补充单元测试。

输出格式：
- 先说明实现思路；
- 再给完整代码；
- 最后说明如何运行和测试。
```

## 9. 代码审查类 Prompt 模板

适合检查 bug、优化复杂度、改进工程质量。

```text
你是一个资深代码审查工程师。

请审查以下代码：
```{language}
{代码}
```

请从以下角度分析：
1. 是否存在 bug；
2. 边界条件是否完整；
3. 时间复杂度和空间复杂度；
4. 命名和代码可读性；
5. 是否存在并发、安全或资源泄漏问题；
6. 是否有更优雅的实现方式。

输出格式：
1. 总体评价；
2. 问题列表；
3. 修改建议；
4. 优化后的代码；
5. 复杂度分析。

要求：
- 不要只说“可以优化”；
- 每个问题都要指出具体位置和原因；
- 修改后的代码要能直接运行。
```

## 10. 总结提炼类 Prompt 模板

适合总结文章、会议记录、课程内容、技术文档。

```text
你是一个擅长信息提炼和结构化总结的助手。

请总结以下内容：
"""
{原文内容}
"""

总结要求：
1. 提炼核心观点；
2. 保留关键事实、数据、结论；
3. 去掉重复和无关内容；
4. 按逻辑结构重新组织；
5. 如果原文有行动项，请单独列出。

输出格式：
## 一句话总结
{一句话概括}

## 核心要点
- {要点1}
- {要点2}
- {要点3}

## 关键细节
| 主题 | 内容 |
| --- | --- |
| {主题} | {内容} |

## 行动项
1. {行动项1}
2. {行动项2}

## 需要进一步确认的问题
- {问题1}
- {问题2}
```

## 11. 研究检索类 Prompt 模板

适合文献调研、资料搜索、竞品分析、技术趋势分析。

```text
你是一个严谨的研究助理，擅长资料检索、来源筛选和结构化综述。

研究主题：
{主题}

研究目标：
{想解决的问题或需要形成的结论}

请完成：
1. 检索并筛选高质量资料；
2. 按主题分类整理；
3. 总结主流观点和代表性方法；
4. 对比不同方案的优缺点；
5. 标注关键信息来源；
6. 给出后续研究方向。

资料要求：
- 优先使用权威来源；
- 优先使用近 {N} 年资料；
- 区分事实、观点和推断；
- 对不确定信息明确说明。

输出格式：
1. 研究结论摘要；
2. 资料分类表；
3. 关键观点总结；
4. 对比分析；
5. 可继续深入的问题。
```

## 12. 决策比较类 Prompt 模板

适合选择专业、岗位、技术路线、项目方案、Offer 对比。

```text
你是一个理性决策顾问，擅长多维度比较和长期收益分析。

我需要在以下选项中做选择：
A：{选项A}
B：{选项B}
C：{选项C，可选}

我的背景和目标：
{个人背景、能力、偏好、长期目标、约束条件}

请从以下维度对比：
1. 短期收益；
2. 长期发展；
3. 学习曲线；
4. 风险和不确定性；
5. 与我当前基础的匹配度；
6. 市场认可度；
7. 未来可迁移性；
8. 推荐优先级。

输出格式：
- 先给明确推荐；
- 再给对比表；
- 然后说明推荐理由；
- 最后给出选择后的行动计划。

要求：
- 不要两边都说好；
- 明确说明适合/不适合的条件；
- 如果存在关键未知信息，请列出，但仍需基于现有信息给出倾向性建议。
```

## 13. Agent / 工具调用类 Prompt 模板

适合需要模型调用工具、查资料、读文件、执行多步任务的场景。

```text
你是一个具备工具使用能力的任务执行 Agent。

最终目标：
{目标}

可用工具：
1. {工具1：用途}
2. {工具2：用途}
3. {工具3：用途}

任务背景：
{背景信息}

执行要求：
1. 先判断任务是否需要使用工具；
2. 如果需要工具，优先使用最可靠的信息来源；
3. 工具结果之间如有冲突，说明冲突并优先采用更权威来源；
4. 不要编造工具没有返回的信息；
5. 每一步都围绕最终目标推进；
6. 输出最终结果前进行自检。

输出格式：
1. 任务理解；
2. 执行过程摘要；
3. 最终结果；
4. 依据来源；
5. 风险或不确定性说明；
6. 下一步建议。
```

## 14. Prompt 优化类 Prompt 模板

适合让模型帮你改写、增强、结构化一个原始 Prompt。

```text
你是一个 Prompt Engineering 专家，熟悉大模型指令设计、上下文组织、输出约束和评估方法。

请优化以下 Prompt：
"""
{原始 Prompt}
"""

优化目标：
1. 任务目标更明确；
2. 上下文边界更清晰；
3. 输出格式更稳定；
4. 约束条件更完整；
5. 减少歧义和无效泛化；
6. 方便后续复用和迭代。

请输出：
1. 优化后的 Prompt；
2. 优化思路说明；
3. 可选增强版本；
4. 使用注意事项。

要求：
- 不要只润色语言；
- 要重构 Prompt 结构；
- 如果原 Prompt 信息不足，请补充占位符；
- 尽量使用可复用模板。
```

## 15. Few-shot 示例模板

当你希望模型稳定模仿某种输出格式时，可以提供示例。

```text
请根据示例格式完成任务。

任务：{任务说明}

示例 1：
输入：{示例输入1}
输出：{示例输出1}

示例 2：
输入：{示例输入2}
输出：{示例输出2}

现在请处理：
输入：
"""
{真实输入}
"""

要求：
- 严格模仿示例的结构；
- 不要添加示例中没有的多余字段；
- 如果输入信息不足，请在对应字段写“信息不足”。
```

## 16. 结构化输出模板

当结果需要被程序解析时，建议明确字段格式。

### JSON 输出模板

```text
请将结果输出为严格 JSON，不要添加 Markdown、解释文字或代码块。

JSON Schema：
{
  "summary": "一句话总结",
  "key_points": ["要点1", "要点2"],
  "risks": ["风险1", "风险2"],
  "next_actions": ["行动1", "行动2"]
}

输入：
"""
{输入内容}
"""
```

### Markdown 表格输出模板

```text
请将结果整理为 Markdown 表格。

表头如下：
| 维度 | 内容 | 说明 | 优先级 |
| --- | --- | --- | --- |

输入：
"""
{输入内容}
"""

要求：
- 每行只表达一个清晰观点；
- 优先级只能填写：高 / 中 / 低；
- 不要输出表格以外的内容。
```

## 17. 自检与质量控制模板

适合放在复杂 Prompt 的最后，用于提高结果可靠性。

```text
在最终回答前，请进行自检：
1. 是否完整完成了任务；
2. 是否严格遵守输出格式；
3. 是否存在没有依据的断言；
4. 是否遗漏了重要约束；
5. 是否有更简单或更可执行的表达；
6. 是否需要标注不确定性。

只输出最终答案，不需要展示自检过程。
```

## 18. Prompt 编写检查清单

写 Prompt 前，可以检查下面的问题：

| 检查项 | 说明 |
| --- | --- |
| 目标是否明确 | 模型是否知道最终要完成什么 |
| 背景是否充分 | 是否提供了必要上下文 |
| 输入是否清晰 | 是否用引号、代码块或分隔符包住输入材料 |
| 约束是否具体 | 是否说明长度、风格、范围、禁止事项 |
| 输出是否指定 | 是否规定 Markdown、JSON、表格、代码等格式 |
| 示例是否需要 | 复杂格式是否提供 few-shot 示例 |
| 质量标准是否明确 | 是否说明什么算好答案 |
| 不确定性如何处理 | 是否要求模型说明不确定内容 |
| 是否可迭代 | 是否方便根据结果继续修改 |

## 19. 常见低质量 Prompt 与改法

### 问题 1：任务太泛

低质量：

```text
帮我分析一下这个项目。
```

更好：

```text
请从业务价值、技术架构、核心模块、实现难点、简历亮点和后续优化方向 6 个角度分析这个项目，并输出 Markdown 表格和改进建议。
```

### 问题 2：没有背景

低质量：

```text
帮我写学习路线。
```

更好：

```text
我有 Java 后端基础，但 Python 和大模型工程经验较少。目标是在 6 个月内具备 Agent 应用开发能力。请按阶段制定学习路线，每阶段包含知识点、实践任务和验收标准。
```

### 问题 3：没有输出格式

低质量：

```text
总结这篇文章。
```

更好：

```text
请按“一句话总结—核心观点—关键证据—可执行建议—疑问点”的结构总结这篇文章。
```

### 问题 4：只要求“详细”，但没有标准

低质量：

```text
详细讲讲 RAG。
```

更好：

```text

## 21. 使用建议

1. 简单任务用“最小可用模板”。
2. 复杂任务用“高质量通用模板”。
3. 需要稳定格式时，加入 few-shot 示例。
4. 需要程序解析时，强制 JSON 或表格结构。
5. 需要可靠性时，加入自检标准和不确定性说明。
6. 需要工具或联网时，明确工具使用条件、来源优先级和引用要求。
7. 不要一次只写“帮我优化一下”，而要说明优化目标、使用场景和输出标准。

## 22. 参考原则摘要

本模板库主要吸收了以下通用原则：

- 指令应放在前面，并用清晰分隔符区分指令和上下文；
- 明确说明任务目标、背景、约束、输出格式和质量标准；
- 用示例展示期望输出，而不只用抽象描述；
- 对复杂任务采用迭代优化，而不是期望一次 Prompt 完美；
- 对需要高可靠性的任务，应提供参考材料、成功标准和自检要求；
- 对工具调用、研究检索、代码任务，应明确工具使用边界、验证方式和异常处理。
